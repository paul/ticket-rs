# Support file and stdin input for text fields via @ convention

Allow text-heavy CLI fields to read content from files or stdin using the `@` prefix convention (same as curl and GitHub CLI). This makes it easy to pass multi-line content, content with special characters (backticks, quotes), and content from other tools via pipes.

## Design

### Syntax

- `@path/to/file.md` — read the field value from a file
- `@-` or `-` — read the field value from stdin
- `@@literal` — escape: value is literally `@literal` (strip one leading `@`)
- Any value not starting with `@` is used as-is (current behavior, unchanged)

### Affected fields

The following flags on `create` and `update` support `@` input:
- `--description` / `-d`
- `--design`
- `--acceptance`

The `add-note` command's positional text argument also supports `@` input. Its existing behavior (read from stdin when text is omitted) is preserved as a fallback.

### Resolution layer

Add a helper function `resolve_input(value: &str) -> Result<String>` that:
1. If `value` starts with `@@`, returns `value[1..]` (strip the escape)
2. If `value` is `-` or `@-`, reads stdin to a string and returns it
3. If `value` starts with `@`, treats `value[1..]` as a file path, reads it, and returns the content (trimming a single trailing newline, matching shell convention)
4. Otherwise returns `value` unchanged

This function lives in a new `src/input.rs` module so it can be shared by `create`, `update`, and `add-note`.

### Stdin conflict detection

Only one field per invocation may read from stdin. If multiple fields resolve to stdin (`-` or `@-`), the command returns an error:

```
Error: only one field may read from stdin at a time
```

This is validated after resolving all inputs but before any ticket I/O.

### Changes to add-note

`add-note` currently reads from stdin when no text argument is provided. With this change:
- `tk note <id> @file.md` — reads note text from file
- `tk note <id> @-` — reads note text from stdin (explicit)
- `tk note <id>` — reads note text from stdin (existing fallback, preserved)
- `tk note <id> "some text"` — inline text (existing, unchanged)

### Error handling

- File not found: `Error: cannot read '@path/to/file.md': No such file or directory`
- File read error: `Error: cannot read '@path/to/file.md': <io error>`
- Multiple stdin: `Error: only one field may read from stdin at a time`

## Acceptance Criteria

- `tk create "Title" --description @desc.md` reads description from file
- `tk create "Title" --design @-` reads design from stdin
- `tk update <id> --acceptance @criteria.md` reads acceptance from file
- `tk note <id> @notes.md` reads note text from file
- `tk note <id>` still reads from stdin when text is omitted (backward compatible)
- `tk create "Title" --description - --design -` errors with stdin conflict message
- `tk create "Title" --description @@github` sets description to literal `@github`
- File-not-found produces a clear error message with the path
- All existing tests pass; new tests cover `resolve_input`, stdin conflict detection, and the `@@` escape

## Implementation Plan

1. Add `src/input.rs` with:
   - `resolve_input(value: &str) -> Result<String>` — resolves `@file`, `@-`/`-`, `@@escape`, or passthrough
   - `validate_no_multiple_stdin(values: &[&str]) -> Result<()>` — checks that at most one value is stdin
2. Add `pub mod input` to `lib.rs`
3. Update `create.rs`: call `resolve_input()` on description, design, acceptance before passing to `create_impl`; validate no multiple stdin
4. Update `update.rs`: same treatment for description, design, acceptance
5. Update `note.rs`: call `resolve_input()` on the text argument when present; preserve stdin fallback when text is `None`
6. Add error variant (or reuse `Error::Io`) for file-read failures with path context
7. Add tests for `resolve_input` (file, stdin, escape, passthrough, errors)
8. Add integration tests for create/update/note with `@` syntax
